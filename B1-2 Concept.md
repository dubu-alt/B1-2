## 1. 메모리 구조 (프로세스 메모리 레이아웃)

### 비유: 아파트 층별 구조

> 프로세스가 사용하는 메모리를 **아파트**라고 생각해보세요.
> 각 층마다 역할이 다릅니다.

```
높은 주소
┌─────────────────────────┐
│         Stack           │ ← 5층: 함수 호출 기록, 지역변수
│    (아래로 성장 ↓)       │   (자동 관리 - 함수 끝나면 사라짐)
├─────────────────────────┤
│           ↕             │ ← 빈 공간 (Stack ↔ Heap이 서로 마주보며 성장)
├─────────────────────────┤
│          Heap           │ ← 4층: 동적 할당 공간 **누수 발생 지점!**
│    (위로 성장 ↑)         │   (수동 관리 - 개발자가 직접 해제해야 함)
├─────────────────────────┤
│          BSS            │ ← 3층: 초기화 안 된 전역변수
├─────────────────────────┤
│          Data           │ ← 2층: 초기화된 전역변수
├─────────────────────────┤
│          Text           │ ← 1층: 실행 코드 자체
└─────────────────────────┘
낮은 주소
```

> **이 문서의 누수는 4층(Heap)에서 발생!**
> 짐을 계속 쌓기만 하고 치우지 않아서 아파트가 꽉 찬 상황

## 2. 💧 메모리 누수 (Memory Leak)

### 비유: 욕조에 물을 받는데 마개를 안 뽑는 것

```
정상적인 메모리 사용:          메모리 누수:

malloc() → 할당 🪣             malloc() → 할당 🪣
사용...                        사용...
free()   → 해제 🚿             free() 없음! ❌

[물 받고 → 빼고 → 반복]        [물 받고 → 받고 → 받고 → 넘침!]
```

### 코드로 보는 차이

```c
// 정상: 할당 후 해제
void 정상함수() {
    char* data = malloc(25MB);  // 할당
    // ... 사용 ...
    free(data);                 // ← 반드시 해제!
}

// ❌ 누수: 해제 없음
void 누수함수() {
    char* data = malloc(25MB);  // 할당
    // ... 사용 ...
    // free(data); ← 이게 없음!
}   // 함수 끝나도 메모리는 그대로 점유 중
```

### 이 문서에서의 패턴

```
T+2초:  malloc(25MB)  free 없음 → 25MB 점유
T+5초:  malloc(25MB)  free 없음 → 50MB 점유
T+8초:  malloc(25MB)  free 없음 → 75MB 점유
T+11초: malloc(25MB)  free 없음 → 100MB 점유 → 종료
```

## 3. OOM (Out of Memory)

### 비유: 주차장이 꽉 찬 상황

> 주차장(메모리)이 꽉 찼는데 새 차(프로세스)가 들어오려 함
> → 관리인(OS)이 강제로 차를 견인(프로세스 종료)

```
OOM 발생 흐름:

메모리 요청
    │
    ▼
사용 가능한 메모리 있음? ──YES──→ 할당 성공 ✅
    │
    NO
    │
    ▼
스왑(Swap) 공간 있음? ──YES──→ 스왑 사용 (느려짐 ⚠️)
    │
    NO
    │
    ▼
OOM Killer 발동! 💀
    │
    ▼
희생 프로세스 선택 → SIGKILL 전송 → 강제 종료
```

### Linux OOM Killer vs 이 문서의 MemoryGuard

| 구분 | Linux OOM Killer | MemoryGuard (이 문서) |
|------|-----------------|----------------------|
| 주체 | OS 커널 | 애플리케이션 자체 |
| 발동 조건 | 시스템 전체 메모리 부족 | 프로세스 한계치 도달 |
| 목적 | 시스템 보호 | 프로세스 자기 보호 |
| 신호 | SIGKILL | SIGKILL (자기 자신에게) |

## 4.프로세스 & 스레드 (Process & Thread)

### 비유: 식당으로 이해하기

```
프로세스 = 식당 전체
┌─────────────────────────────────┐
│           식당 (Process)         │
│  ┌──────┐ ┌──────┐ ┌──────┐   │
│  │웨이터1│ │웨이터2│ │웨이터3│   │ ← 스레드들
│  │Thread│ │Thread│ │Thread│   │
│  └──────┘ └──────┘ └──────┘   │
│                                 │
│  [공유 주방 = 공유 메모리(Heap)] │ ← 모든 스레드가 같은 Heap 사용!
└─────────────────────────────────┘
```

> **이 문서의 위험 포인트**
> `MULTI_THREAD_ENABLE=True` 상태에서
> 여러 스레드가 **같은 Heap**에 동시 접근
> → 메모리 누수가 더 빠르게 진행될 수 있음

### 멀티스레드 환경의 추가 위험

```
Thread 1: malloc(25MB) ──┐
Thread 2: malloc(25MB) ──┼──→ Heap 동시 접근
Thread 3: malloc(25MB) ──┘    (동기화 없으면 더 위험!)

경고 메시지가 나온 이유:
"POTENTIAL DEADLOCK IN CONCURRENT MODE"
→ 스레드들이 서로 자원을 기다리다 멈출 수 있음
```

## 5. 데드락 (Deadlock)

### 비유: 좁은 골목에서 마주친 두 차

```
Thread A가 가진 것: 자원 🔑1
Thread A가 필요한 것: 자원 🔑2

Thread B가 가진 것: 자원 🔑2
Thread B가 필요한 것: 자원 🔑1

결과:
Thread A: "🔑2 줘!" → Thread B: "🔑1 먼저 줘!"
Thread B: "🔑1 줘!" → Thread A: "🔑2 먼저 줘!"

→ 둘 다 영원히 대기... 💀 데드락!
```

### 데드락 발생 4가지 조건 (모두 만족해야 발생)

```
1. 상호 배제 (Mutual Exclusion)
   → 자원을 한 번에 하나의 스레드만 사용 가능

2. 점유 대기 (Hold and Wait)
   → 자원을 가진 채로 다른 자원을 기다림

3. 비선점 (No Preemption)
   → 강제로 자원을 빼앗을 수 없음

4. 순환 대기 (Circular Wait)
   → A→B→C→A 형태로 순환하며 대기
```

## 6. 선형 증가 패턴 분석

### 이 문서에서 관찰된 패턴의 의미

```
메모리(MB)
100 |                    ●  ← 종료
 75 |               ●
 50 |          ●
 25 |     ●
  0 +----+----+----+----+
    0    3    6    9   (초)

기울기 = 25MB / 3초 ≈ 8.3MB/초
→ 일정한 기울기 = 규칙적인 누수
→ 랜덤하지 않음 = 의도된 동작 or 반복 루프
```

> **패턴으로 알 수 있는 것**
> - **랜덤 누수**: 불규칙한 증가 → 버그성 누수
> - **선형 누수 (이 경우)**: 규칙적 증가 → 반복 루프에서 해제 없이 할당

```python
# 추정되는 내부 동작 (의사 코드)
def memory_worker():
    while True:
        data = allocate(25 * MB)  # 할당
        # free(data) ← 이게 없음!
        time.sleep(3)             # 3초 대기
```

## 7. 시그널 (Signal) - SIGKILL

### 비유: 프로세스에게 보내는 메시지

```
Signal 종류:
┌──────────┬──────────────────────────────────────────┐
│ SIGTERM  │ "정리하고 종료해줘" (부드러운 요청)        │
│ SIGKILL  │ "즉시 종료!" (거부 불가, OS가 강제 실행)   │
│ SIGINT   │ "Ctrl+C" (인터럽트)                       │
│ SIGSEGV  │ "잘못된 메모리 접근!" (세그폴트)           │
└──────────┴──────────────────────────────────────────┘

이 문서:
MemoryGuard → SIGKILL → 프로세스 ID 7 즉시 종료
                ↑
         핸들러 등록 불가
         = 프로세스가 막을 방법 없음
```

## 8. 가비지 컬렉션 (Garbage Collection)

### 비유: 자동 청소부 vs 수동 청소

```
언어별 메모리 관리 방식:

C/C++ (수동)          Python/Java (자동 GC)
개발자가 직접 청소     청소부(GC)가 알아서 청소

malloc() → 할당       new Object() → 할당
...사용...            ...사용...
free()   → 해제 ✋    // 자동으로 해제됨 🤖
          ↑
       잊으면 누수!
```

### GC가 있어도 누수가 생기는 경우

```python
# Python에서도 누수 가능한 패턴!
cache = []  # 전역 리스트

def 문제있는함수():
    data = [0] * (25 * 1024 * 1024)  # 25MB
    cache.append(data)  # 전역에 추가
    # GC가 있어도 cache가 참조 중이라 해제 못 함!
    # → 참조가 남아있으면 GC도 수거 불가
```

> **GC의 한계**: 참조(Reference)가 남아있으면 수거 불가
> → 이 문서에서 "가비지 컬렉션 개선" 권장사항이 나온 이유

## 02번 (CPU 분석) - 필수 개념

### 1. CPU 스케줄링 (CPU Scheduling)

**정의**: 여러 프로세스가 CPU를 공유할 때, 어떤 프로세스가 언제 CPU를 사용할지 결정하는 것

**현실적 예시**:
```
1초 동안 4개의 프로세스가 있다면:
- Process A: 0.0 ~ 0.25초 CPU 사용
- Process B: 0.25 ~ 0.5초 CPU 사용
- Process C: 0.5 ~ 0.75초 CPU 사용
- Process D: 0.75 ~ 1.0초 CPU 사용

→ OS가 프로세스들을 번갈아가며 실행
```

**우리 테스트에서**:
```
CpuWorker가 5% ~ 30% CPU 사용
= OS가 CpuWorker에게 CPU 시간을 일부 할당
= 나머지 CPU는 다른 프로세스에 할당
```

---

### 2. CPU 사용률 (CPU Utilization)

**정의**: CPU가 실제로 작업하는 시간의 비율

**계산**:
```
CPU 사용률 = (실제 작업 시간 / 총 시간) × 100%

예:
- 1초 동안 0.3초 작업 → 30% CPU 사용률
- 1초 동안 0.05초 작업 → 5% CPU 사용률
```

**로그에서 보이는 것**:
```
2026-05-27 05:58:52,265 [INFO] [CpuWorker] Current Load: 5.00%
→ 지난 1초 동안 CPU를 5% 사용했다는 뜻
```

---

### 3. CPU 한계 (CPU Limit / Threshold)

**정의**: 특정 프로세스가 사용할 수 있는 최대 CPU 비율

**우리 테스트**:
```
CPU_MAX_OCCUPY=30%
= CpuWorker는 최대 30%까지만 CPU 사용 가능
= 30%를 초과하면 제한됨
```

**현실적 예시**:
```
컴퓨터 CPU가 100% 능력이라고 하면:
- 이 프로세스: 최대 30까지만 사용
- 다른 프로세스들: 나머지 70을 나눠 씀
```

---

### 4. Throttling (스로틀링 / 조절)

**정의**: CPU 사용을 의도적으로 줄이는 행위

**우리 로그에서**:
```
2026-05-27 05:59:12,998 [INFO] [CpuWorker] Peak reached (30.00%). Starting cooldown...
```
→ CPU가 30%에 도달하자 CPU 사용을 줄임 (throttle)

**실생활 예시**:
```
- 노트북 배터리 부족 → CPU 속도 낮춤
- 폰 발열 높음 → CPU 성능 제한
- 서버 부하 높음 → 특정 서비스 속도 제한
```

---

### 5. Cooldown (쿨다운 / 회복 시간)

**정의**: 과부하 상태에서 벗어나기 위해 대기하는 기간

**우리 테스트에서**:
```
Phase 1: CPU 상승 (5% → 30%)
         ↓
Phase 2: Cooldown 시작 (Peak reached)
         CPU를 30% → 5%로 낮춤
         약 16초 소요
         ↓
Phase 3: Cooldown 완료 (5%)
         다시 상승 시작
```

**시간대**:
```
05:59:12 - Cooldown 시작
05:59:28 - Cooldown 완료 (약 16초)
```

---

### 6. 워크로드 (Workload)

**정의**: 프로세스가 해야 할 일의 양 또는 강도

**우리 테스트**:
```
CpuWorker의 워크로드:
- 초기: 낮음 (5% CPU만 사용)
- 중간: 증가 (점점 더 많은 계산)
- 최고: 30% (최대 허용치)
- 쿨다운: 낮춤 (의도적으로 일을 줄임)
```

## 03번 (Deadlock 분석) - 필수 개념

### 1. 멀티스레드 (Multithreading)

**정의**: 하나의 프로세스 내에서 여러 개의 스레드가 동시에 실행되는 것

**프로세스 vs 스레드**:
```
프로세스 (Process):
- 독립적인 프로그램 실행 단위
- 각자 메모리 공간 보유
- 예: 크롬, 유튜브, 카톡 앱

스레드 (Thread):
- 같은 프로세스 내의 실행 단위
- 메모리 공간 공유
- 예: 크롬 내의 여러 탭들이 동시에 작동

우리 테스트:
agent-leak-app 프로세스 내에
├─ Worker-Thread-1
├─ Worker-Thread-2
└─ (다른 스레드들)
이 동시에 실행
```

---

### 2. 자원 (Resource)

**정의**: 여러 스레드가 공유하는 데이터나 객체

**우리 테스트의 자원**:
```
1. Shared_Memory_A (공유 메모리)
2. Socket_Pool_B (소켓 풀)

이 둘을 여러 스레드가 동시에 접근하려 함
```

**현실적 예시**:
```
은행의 계좌 잔액 (자원):
- 스레드1: 입금 시도
- 스레드2: 출금 시도
→ 동시에 접근하면 일관성 깨짐
→ 따라서 한 번에 1개 스레드만 접근 가능
```

---

### 3. 락 (Lock / Mutual Exclusion - 상호 배제)

**정의**: 한 번에 1개 스레드만 자원을 사용하도록 제한하는 메커니즘

**동작**:
```
Lock 획득:
스레드1: "Shared_Memory_A 사용하겠습니다" (Lock 획득 ✓)
스레드2: "Shared_Memory_A 사용하겠습니다" (Lock 대기... ⏳)

스레드1이 작업 완료:
스레드1: "다 했습니다" (Lock 해제)
스레드2: "이제 사용합니다" (Lock 획득 ✓)
```

**우리 로그에서**:
```
2026-05-27 05:50:08,816 [INFO] LOCK ACQUIRED: [Shared_Memory_A]
→ 이 스레드가 락을 획득함
→ 다른 스레드는 접근 불가
```

---

### 4. 교착상태 (Deadlock)

**정의**: 2개 이상의 스레드가 서로가 가진 자원을 기다리며 무한히 대기하는 상태

**우리 테스트의 상황**:
```
시간순서:
T1: 스레드1이 자원A 획득
    스레드2가 자원B 획득

T2: 스레드1이 자원B 필요 → 스레드2가 가지고 있음 → 대기
    스레드2가 자원A 필요 → 스레드1이 가지고 있음 → 대기

T3: 둘 다 자신이 필요한 자원을 못 가지고 있음
    → 영원히 대기 (교착상태)

로그:
2026-05-27 05:50:10,820 [INFO] WAITING for [Socket_Pool_B]... (Status: BLOCKED)
2026-05-27 05:50:10,820 [INFO] WAITING for [Shared_Memory_A]... (Status: BLOCKED)
→ 둘 다 BLOCKED 상태에서 멈춤
```

**현실적 예시**:
```
교과서의 "식사하는 철학자 문제":

철학자 A, B가 앉아있고 포크 2개가 있음
(A 왼쪽에 포크1, 오른쪽에 포크2, B 오른쪽에 포크1... 등)

A: 왼쪽 포크 집음 (포크1 획득)
B: 오른쪽 포크 집음 (포크2 획득)

A: 오른쪽 포크 필요 → B가 가지고 있음 (대기)
B: 왼쪽 포크 필요 → A가 가지고 있음 (대기)

→ 둘 다 영원히 기다림 (밥을 못 먹음)
```

### 5. 데드락 4가지 필수 조건 (Deadlock Conditions)

데드락이 발생하려면 **다음 4가지 조건을 모두 만족**해야 함:

#### 조건 1: 상호 배제 (Mutual Exclusion)
```
자원이 배타적으로 사용됨
= 한 번에 1개 스레드만 사용 가능

우리 경우:
Shared_Memory_A는 한 번에 1개 스레드만 접근 가능
Socket_Pool_B도 한 번에 1개 스레드만 접근 가능

✓ 만족
```

#### 조건 2: 점유 대기 (Hold and Wait)
```
스레드가 자원을 보유하면서 다른 자원 대기
= 가진 것 놓지 않고 필요한 것 기다림

우리 경우:
스레드1: 자원A 보유 → 자원B 대기
스레드2: 자원B 보유 → 자원A 대기

✓ 만족
```

#### 조건 3: 비선점 (No Preemption)
```
자원을 강제로 빼앗을 수 없음
= 획득한 스레드만 해제 가능

우리 경우:
스레드1이 자원A를 가지면
다른 스레드가 강제로 빼앗을 수 없음
(스레드1이 해제할 때까지 대기)

✓ 만족
```

#### 조건 4: 순환 대기 (Circular Wait)
```
자원 대기가 순환 구조를 이룸
= A → B → A 형태 (또는 A → B → C → A 등)

우리 경우:
스레드1 (자원A 보유) → 스레드2 (자원B 보유) → 스레드1
                    ↑                              ↓
                    └──────────────────────────────┘

순환 구조!

✓ 만족
```

**결론**: 4가지 조건 모두 만족 → **데드락 발생**

### 6. 자원 획득 순서 (Resource Ordering)

**정의**: 여러 자원을 사용할 때 획득하는 순서

**문제 상황 (우리 테스트)**:
```
스레드1: 자원A → 자원B 순서로 획득
스레드2: 자원B → 자원A 순서로 획득
                 ↑
                다른 순서!
```

**해결책**:
```
모든 스레드가 동일 순서로 획득:

방법1:
스레드1: 자원A → 자원B
스레드2: 자원A → 자원B (동일 순서)

방법2:
스레드1: 자원B → 자원A
스레드2: 자원B → 자원A (동일 순서)

→ 순환 대기 불가능!
```

### 7. 블로킹 (Blocking) vs 대기 (Waiting)

**블로킹 (Blocking)**:
```
스레드가 자원을 기다리며 실행을 멈춤

로그에서:
[INFO] WAITING for [Socket_Pool_B]... (Status: BLOCKED)
→ BLOCKED 상태 = 실행하지 않음, 기다리는 중
```

**CPU/메모리 상태**:
```
BLOCKED 상태의 스레드:
- CPU: 0% (실행하지 않음)
- 메모리: 변화 없음 (새로 할당 안 함)
- 로그: 안 나옴 (I/O 수행 안 함)
```

**우리 테스트 결과**:
```
2026-05-27 05:50:10,820 이후:
- CPU: 0%
- 메모리: 정체
- 로그: 없음
→ 모든 스레드가 BLOCKED 상태
```

| 항목 | CPU (02번) | Deadlock (03번) |
|------|-----------|-----------------|
| **문제** | CPU 과사용 | 자원 경쟁 |
| **해결책** | Cooldown (자동 조절) | 자원 순서 통일 |
| **프로세스 상태** | 계속 실행 (CPU 낮춤) | 멈춤 (BLOCKED) |

| **로그** | 계속 나옴 | 멈춤 |
| **자동 복구** | ✓ 있음 | ✗ 없음 |
| **위험도** | 낮음 (조절됨) | **높음** (무한 대기) |

## 여러 개념들

### CPU 관련
1. **CPU 스케줄링**: 여러 프로세스가 CPU를 공유하는 방식
2. **CPU 사용률**: (실제 작업시간 / 총시간) × 100%
3. **Throttling**: CPU 사용을 의도적으로 제한
4. **Cooldown**: 과부하에서 벗어나기 위한 회복 기간
5. **Workload**: 프로세스가 해야 할 일의 양

### Deadlock 관련
1. **Multithreading**: 하나의 프로세스 내 여러 스레드 동시 실행
2. **자원**: 여러 스레드가 공유하는 데이터/객체
3. **락(Lock)**: 한 번에 1개 스레드만 접근하도록 제한
4. **Deadlock**: 2개 이상 스레드가 서로의 자원을 무한히 기다리는 상태
5. **데드락 4가지 조건**: Mutual Exclusion, Hold and Wait, No Preemption, Circular Wait
6. **자원 획득 순서**: 모든 스레드가 동일 순서로 자원 획득

3. 한계에 도달하면? (= Cooldown으로 자동 조절)
4. 프로세스는? (= 계속 살아있음)

**Deadlock 이해하기**:
1. 스레드가 여럿이면? (= Multithreading)

3. 서로 다른 순서로 획득하면? (= Deadlock 위험!)
4. 한 스레드가 멈추면? (= 다른 스레드도 멈춤)

## 요약

### CPU (02번)
```
CpuWorker가 부하를 계속 증가시킴
→ 30%에 도달하면 "Peak reached"
→ Cooldown 시작 (부하 자동 감소)
→ 5%까지 내려가면 "Cooldown complete"
→ 다시 부하 증가
→ 반복 (무한 사이클)

결론: CPU 정상 작동, 자동 조절 메커니즘 우수
```

### Deadlock (03번)
```
스레드1: 자원A 획득 → 자원B 대기
스레드2: 자원B 획득 → 자원A 대기

→ 서로의 자원을 무한히 기다림
→ 4가지 데드락 조건 모두 만족
→ 프로세스 멈춤 (응답 불가)

결론: 위험! 즉시 MULTI_THREAD=false로 전환 필요
```
